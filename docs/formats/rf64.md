# RF64 / BW64 Format Internals

Low-level reference for RF64 (EBU Tech 3306) and BW64 (ITU-R BS.2088) files:
the 64-bit extension of RIFF/WAVE for audio larger than 4 GB.

---

## Overview

RIFF/WAVE stores every chunk size as a `uint32`, capping a file at 4 GB. RF64
lifts that cap: the magic becomes `RF64` (or `BW64`), the 32-bit size fields
that would overflow are set to the sentinel `0xFFFFFFFF`, and a mandatory `ds64`
chunk carries the true 64-bit sizes. Everything else is ordinary WAVE.

acidcat walks RF64 natively on the `inspect` path, resolving the sentinels via
`ds64`. (The index-path sniffer only matches `RIFF....WAVE`, so an RF64 file
sniffs as None there and falls back to extension dispatch, see
format_internals.md.)

---

## File Structure

```
+-------------------------------------------+
| "RF64" 0xFFFFFFFF "WAVE"                   |  32-bit riff size is a sentinel
+-------------------------------------------+
| ds64 (size)                               |  the 64-bit sizes live here
|   riff_size, data_size, sample_count (u64)|
|   + chunk-size override table             |
+-------------------------------------------+
| fmt  ...                                  |
| data 0xFFFFFFFF ...                        |  32-bit data size is a sentinel
+-------------------------------------------+
```

### Header + ds64

```
offset  size  field
0       4     "RF64" or "BW64"
4       4     riff_size32   = 0xFFFFFFFF sentinel (real size in ds64)
8       4     "WAVE"
12      4     "ds64"
16      4     ds64 body size (uint32 LE)
20      8     riff_size     uint64 LE: true total size
28      8     data_size     uint64 LE: true data-chunk size
36      8     sample_count  uint64 LE: samples per channel
44      4     table_length  uint32 LE: number of override entries
48      ...   table[]       { chunk id[4], size uint64 } per oversized chunk
```

`ds64` must be the first chunk. Any chunk whose 32-bit size is the
`0xFFFFFFFF` sentinel takes its real size from `data_size` (for `data`) or the
override table (for anything else).

---

## acidcat inspect

`acidcat inspect FILE` (an RF64 `.wav`) renders the same chunk set as RIFF/WAVE,
plus:

- **ds64** , decoding `riff_size`, `data_size`, `sample_count`, the
  `table_length`, and each override-table entry.
- the **data** chunk with its `0xFFFFFFFF` size resolved to the `ds64`
  `data_size`, so the derived duration and frame count are correct.

The walker warns if `ds64` is missing (an RF64 file that cannot resolve its own
sentinels) or if the resolved sizes disagree with the file length.

---

## Notes

- Little-endian throughout (it is RIFF); only the widths change (u64 in `ds64`).
- `BW64` is the ITU name for the same layout; treat the two magics identically.
- The override table is why RF64 needs its own walker rather than a size patch:
  any chunk, not just `data`, can be oversized and deferred to the table.
- A well-formed small RF64 is legal (the sentinels and `ds64` are structural,
  not size-gated), which is what makes a compact test specimen possible.
