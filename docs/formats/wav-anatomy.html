A WAV is a RIFF container: the 12-byte RIFF header, then a run of chunks, each an id, a 32-bit little-endian size, and that many payload bytes (plus a pad byte if the size is odd). But WAV is one branch of a wide family — from the 1985 Amiga IFF it descends from, to the broadcast and large-file variants (BWF, RF64, Wave64) that outgrew its 4 GB ceiling, to its big-endian sibling AIFF. The chunked container is not a WAV invention. It descends from Electronic Arts' IFF (1985, Amiga): a FORM whose body is a run of chunks, each a 4-character id + a 32-bit size + payload, padded to an even boundary — so a reader that meets an unknown chunk knows exactly how many bytes to skip. Microsoft's RIFF (1991) is that grammar with the byte order reversed for the little-endian PC; its designer Greg Riker had co-authored IFF, and its name means “Resource IFF / Reverse IFF.” WAVE is the RIFF form for audio. Everything else on these tabs is a dialect layered on that one idea.

the family, dated
1985EA IFF 85 — the ancestor. FORM/LIST/CAT, big-endian 4CC+size chunks, even-byte pad.
1988AIFF (Apple) — IFF for audio, big-endian, COMM + SSND, 80-bit sample rate. The sibling, not the child.
1991RIFF / WAVE (Microsoft + IBM) — IFF reversed to little-endian for the PC. fmt + data.
1996BWF (EBU Tech 3285) — the bext chunk: originator, timestamp, coding history. + Wave64 GUIDs minted (Sony).
1999Wave64 (Sonic Foundry) finalized — 128-bit GUID chunk ids, 64-bit sizes, to break 4 GB.
2007RF64 (EBU Tech 3306) — the ds64 chunk carries 64-bit sizes; magic flips to RF64 only when needed.
2019BW64 (ITU-R BS.2088) — RF64 generalized, tied to the Audio Definition Model (chna/axml).
the shared grammar. Every format on these tabs is the same atom: a 4-byte id, a size, a payload, and a pad byte when the size is odd. Only three things vary across the family — the byte order (IFF/AIFF big-endian, RIFF little-endian), the width of the size field (32-bit, or 64-bit in RF64/Wave64), and whether the id is a 4-character code or a 16-byte GUID (Wave64). Learn the atom once and every dialect reads the same.
the three normative WAVE rules. Of everything a writer can do, only these are guaranteed: the file opens with RIFF (or RF64); the form type is WAVE; and fmt precedes data. Chunk order is otherwise free, the declared RIFF size is often stale, and acidcat lints every claim against the bytes that are actually present rather than trusting the header.

magic"RIFF" / "RIFX"sizeu32, 4 GB cap1991MS + IBM

The container itself: 12 bytes that every WAV opens with. "RIFF", a 32-bit little-endian size, and a form type "RIFF" wraps IFF's grammar with the byte order flipped for the Intel PC — the little-endian concession that gave it the name Reverse IFF. A rare big-endian variant, RIFX, keeps everything else the same and flips only the number encoding. The riff_size field claims the file length minus 8, but writers routinely leave it stale, so acidcat lints it rather than trusting it.

RIFF header (12 bytes)

Example: a small WAVE file, 2,084 bytes total.

RIFX — big-endian RIFF. Identical layout, but the size fields and all multi-byte values are big-endian, and the magic is "RIFX". Microsoft defined it; almost nothing writes it. Macromedia Director used big-endian RIFF internally — whether that was registered RIFX or an independent reinvention is unrecorded.
other RIFF forms. WAVE is one of many form types after the size field: AVI (video), RMID (RIFF-wrapped MIDI), RDIB (bitmap), PAL (palette), RMMP (Director), sfbk (SoundFont), DLS (Downloadable Sounds). The form type at 0x08 is what a reader branches on.
tag0xFFFEfmt size40 bytessub-format16-byte GUID

The modern fmt descriptor. When the format tag is 0xFFFE, the fmt chunk grows to 40 bytes: the 16-byte PCM core, then a cb_size, then three extension fields — the real bit depth (valid_bits, so 24-in-32 padding is explicit), a channel mask naming which speakers each channel drives, and a 16-byte sub-format GUID whose leading 2 bytes hold the actual codec tag and whose fixed tail identifies it as a standard KSDATAFORMAT subtype. This is how multichannel and high-bit-depth PCM are described unambiguously.

magic"RF64"sizesu64 in ds642007EBU Tech 3306

The large-file answer that stays RIFF. A file over 4 GB flips its magic from RIFF to RF64, writes 0xFFFFFFFF as a sentinel in the 32-bit size fields, and adds a ds64 chunk as the first chunk carrying the real 64-bit sizes. A clever writer reserves a JUNK chunk of exactly the ds64 size up front while the file is small, then overwrites it in place with ds64 if the recording crosses 4 GB — so the file never has to be rewritten. (That JUNK-to-ds64 reservation, at 28 bytes right after fmt, is itself an ffmpeg fingerprint acidcat reads.)

ds64 fixed area (28 bytes)

Example: a 4.3 GB recording — just past the u32 ceiling, which is the whole point.

the override table. After the 28-byte fixed area, table_length records follow, each a 4-byte chunk id + a u64 size — for any non-data chunk that also carries the 0xFFFFFFFF sentinel. Usually zero: only the data chunk is large. A sentinel with no ds64 override is a corruption acidcat lints.
BW64. ITU-R BS.2088 generalizes RF64 as BW64 (magic "BW64"), adding the Audio Definition Model via the chna and axml chunks for object/immersive audio. The ds64 mechanism is unchanged.
ids16-byte GUIDssizesu64SonySonic Foundry

Sony's take on the 4 GB problem, and the odd one out. Wave64 abandons 4-character codes entirely: every chunk id is a 128-bit GUID and every size is a 64-bit value that includes the 24-byte chunk header (unlike RIFF, which counts only the payload). No original Sonic Foundry spec is known to survive — the format is reconstructed from libsndfile and from the GUIDs themselves, which are version-1 UUIDs carrying a timestamp: the container GUIDs date to April 1996, the audio-chunk GUIDs to December 1999.

Wave64 header (40 bytes)

The RIFF GUID + 64-bit file size + WAVE GUID. Watch the first four bytes of each GUID.

the GUID trick. The first four bytes of each Wave64 GUID are literally the little-endian ASCII of the old 4CC — 72 69 66 66 spells riff, 77 61 76 65 spells wave. The remaining twelve bytes are a fixed v1-UUID suffix. Decoding that suffix's embedded timestamp is what dates the format; the container GUIDs share their node bytes with Microsoft DirectShow GUIDs of 1996-97, and Sonic Foundry's CTO was ex-Microsoft.
magic"FORM" .. "AIFF"endianbig1988Apple

The mirror image. AIFF is IFF-for-audio done by Apple in 1988 — the same FORM/chunk grammar as RIFF, but big-endian throughout (the 68k Mac's byte order, the one IFF kept and RIFF reversed). Its COMM chunk is the analogue of fmt, and it stores the sample rate in the strangest field in the whole family: an 80-bit IEEE-754 extended-precision float, a format Python's struct cannot unpack, which acidcat reassembles by hand.

COMM chunk (18 bytes)

Example: 16-bit stereo at 44100 Hz, 88200 frames.

the 80-bit rate. Where WAV writes the sample rate as a plain u32, AIFF uses a 10-byte extended float: 1 sign bit, a 15-bit exponent biased by 16383, and an explicit 64-bit mantissa (no hidden leading 1, unlike float64). 44100 encodes as 40 0E AC 44 00 00 00 00 00 00. An all-ones exponent (IEEE inf/NaN) is treated as unset, never a rate.
