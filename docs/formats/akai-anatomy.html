An .akp is a keygroup program for the Akai S5000/S6000 samplers — the map that tells the sampler which sample to play for each key and velocity. It is a RIFF file (form type APRG) of little-endian IFF chunks: a prg header, then out, tune, lfo, mods, and one kgrp per keygroup. Each kgrp is itself nested IFF — kloc (key range), three env, a filt, and up to four zone chunks, each naming the sample it plays. The audio itself lives in sibling .wav files, so the program holds references, not samples. The tabs open each structure, drawn byte by byte from a real S6000 program. The Akai program is pure mapping: it carries no audio, only the rules that turn a MIDI note into a sample. The file is a RIFF of form type APRG, and the interesting part is that it nests — the top level lists program-wide chunks (prg, out, tune, lfo, mods), and then a kgrp chunk per keygroup that contains its own chunk stream (kloc, env, filt, zone). Each zone names an external .wav. The program below is a real S6000 acoustic-guitar patch.

the S-series, dated
1988Akai S1000 — the studio-standard hardware sampler of its era. Its program/keygroup model (a key map of velocity-split zones) is the ancestor of the .akp layout.
1998S5000 / S6000 — the generation that saved programs to disk as .akp (RIFF/APRG), samples as .wav, and multis as .akm. A three-file split: mapping, audio, and arrangement.
keygroupthe unit of an Akai program. A keygroup owns a key range and up to four velocity zones; the sampler picks the zone by how hard the key is struck. Layering four sampled velocities per key is how the S-series got its realism.
nowread by converters. Chicken Systems Translator, Extreme Sample Converter and others still parse .akp to migrate S-series libraries into soft samplers. acidcat surfaces the map and the referenced sample names.
nested IFF, not flat. Most RIFF files are one level deep. An Akai program is two: a kgrp chunk's payload is itself a stream of IFF chunks (kloc, env, filt, zone). acidcat walks the outer chunks, then re-enters each kgrp to walk its inner ones — the same IFF reader, recursed one level.
RIFF size is zero. Akai writes 0x00000000 for the RIFF size field rather than the real body length — a firmware quirk. A strict RIFF reader that trusts the size reads nothing; acidcat falls back to the actual file length and walks the chunks regardless. It is one of those places a real specimen corrects a spec.
the samples are elsewhere. A zone stores a sample name, not audio — the S6000 loads the matching .wav from the same volume. So acidcat treats the zones as references: it lists which samples a program needs (here "12 STRING G5"), and a missing sibling .wav is a broken program, not a broken file.

The file opens like any RIFF — a RIFF magic, a size, a form type — but the form type is APRG (Akai program) and the size field is left zero. The first chunk, prg, is the program header: its bytes carry the MIDI program number and the declared keygroup count, which acidcat cross-checks against the number of kgrp chunks it actually finds. trust the file, not the size. Because riff_size is 0, acidcat sizes the walk from the actual file length and reads chunks until it runs out — the same degrade-gracefully rule the WAV walker uses for a stale RIFF size. Every Akai program does this, so it is expected, not an error.
prg declares, kgrp confirms. The prg body's third byte is the keygroup count (9 here). acidcat compares it to the number of kgrp chunks it walks; a mismatch is warned, catching a truncated or hand-edited program. The second byte is the MIDI program number the sampler assigns.

A kgrp chunk is one keygroup, and its payload is a whole IFF stream of its own. The first inner chunk is kloc — the key location: the low and high MIDI notes this keygroup answers to. After it come three env (amplitude/filter/aux envelopes), a filt, and the zone chunks. The map is the start of a real kgrp, showing its first inner chunk, kloc.

kgrp payload, opening kloc chunk (16 bytes)

Hover low_key / high_key — this keygroup covers MIDI notes 88-127.

a keygroup is a key window plus velocity layers. kloc sets the window (low..high MIDI note); the four zone chunks that follow are the velocity layers within it. A program spreads keygroups across the keyboard, each a slice of the range — nine of them here, covering the guitar top to bottom.
the same walker, one level down. There is nothing special about the inner chunks: they are ordinary IFF (4-byte id, little-endian u32 size, payload, word-padded). acidcat re-enters the kgrp body with the same chunk iterator it used at the top level — recursion, not a second parser.

A zone is one velocity layer of a keygroup, and its most important content is a sample name: the .wav the sampler plays when a key in this group is struck in this zone's velocity band. The name is stored length-prefixed — a flag byte, a length, then the ASCII — followed by the zone's tuning and level parameters. Up to four zones stack per keygroup; unused ones carry an empty name. The map is a real zone naming "12 STRING G5".

zone chunk (id + size + flag + name)

Hover sample_name — the external .wav this zone references.

length-prefixed, not null-terminated. The zone names its sample as [flag][length][bytes] — byte 1 is the name length, and exactly that many ASCII bytes follow. A length of 0 means an empty zone (no sample for that velocity band); acidcat skips those and lists only the real references.
this is the reference graph edge. The zone is where an Akai program points outward: "12 STRING G5" is a filename the S6000 resolves on the same volume. acidcat collects every distinct zone name as the program's sample dependency list — the set of .wavs you need alongside the .akp for it to make sound.
