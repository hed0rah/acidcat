Architecture

Per-library DB
SQLite + WAL + FTS5
One DB per registered directory. Schema 1: samples / scan_roots / tags / descriptions / features / samples_fts.
Global registry
~/.acidcat/registry.db
Single SQLite file listing every library plus cached counts and last-indexed timestamps.
Fan-out query
read across all libs
Open every registered DB, run the same WHERE on each, dedup by path, sort, apply final LIMIT.
Longest-prefix route
writes target one lib
find_library_for_path picks the most specific registered root containing a given sample path.
MCP cost prefix
Fast / SLOW / VERY SLOW
Tool descriptions lead with a budget prefix the LLM uses for ranking. Destructive prefix not yet wired (see F-17).
Camelot wheel
key compatibility
Maps musical keys to the 1A-12B / 1B-12B wheel; compatible_keys returns the wheel-adjacent set used by find_compatible.

Recommended fix order

  1. One-line annotation / description fixes. No risk. F-04 F-09 F-17 F-20 F-24 F-29. free wins
  2. Pre-PyPI bug fixes. Real correctness gaps the wider audience would hit. F-01 F-05 F-07 F-13 F-23. before public release
  3. Behavior-changing post-PyPI items. Need release notes. F-02 F-06 F-08 F-14 F-15 F-18 F-22. v0.6.x
  4. Polish. Nits, edge cases, hardening. F-10 F-12 F-16 F-21 F-25 F-26 F-27 F-28. when convenient

Non-findings (clean surface)

  • SQL injection: all dynamic SQL uses parameterized placeholders. The only dynamic-text path is FTS5 which has its own non-SQL syntax issue (F-07).
  • Path traversal: MCP tools route through the registry. Arbitrary file paths cannot escape registered library roots.
  • Command injection: no shell-out anywhere in production code. Subprocess use is in test fixtures only.
  • Secret exposure: no creds, tokens, or API keys in tree.
  • Dependency surface: minimal. sqlite3 stdlib, mutagen, optionally librosa+numpy+scikit-learn.
  • Test coverage: 232 tests across 10 files exercise core flows: index, query, registry, FTS, format parsers, MCP fan-out, Camelot.
