# acidcat v0.5.0 field notes

Real-world observations from using v0.5.0 on actual sample libraries.
Captured as we go so v0.5.1 has a real evidence base instead of guesses.

## Bugs found

### BPM parser misfires on `<digits><letter>` filenames

**Symptom**: filenames like `91V_SBH_126_drum_fill_build_up_dont_stop.wav`
get indexed with `bpm=91.0` instead of the correct `bpm=126.0`. The
`91V` is a pack identifier (probably "91 Vibes"), not a tempo.

**Cause**: the bare-number BPM pattern in
`core/detect.py::parse_bpm_from_filename` is

```python
r'(?<![\d.])(\d{2,3})(?![\d.])'
```

Zero-width lookarounds reject other digits and decimals adjacent to the
number, but they accept letters. So `91V` matches because "V" is
neither a digit nor a `.`. The parser returns the first in-range match
(`91`) and never reaches `_126_`.

**Fix**: tighten the lookarounds to also reject letters.

```python
r'(?<![\w.])(\d{2,3})(?![\w.])'
```

`\w` is `[A-Za-z0-9_]` so we also need the underscore to remain a valid
delimiter. `(?<![\d.a-zA-Z])(\d{2,3})(?![\d.a-zA-Z])` is the explicit
form: number boundaries must be non-alphanumeric, non-decimal,
non-letter. Underscores, hyphens, spaces, slashes still work.

**Test cases the fix needs to cover**:
- `91V_SBH_126_drum.wav` -> 126 (was 91)
- `_91_drum.wav` -> 91 (still works)
- `BPM91_drum.wav` -> 91 (still works via the `(\d{2,3})bpm` and bpm-prefix patterns)
- `Pack91_drum_126_loop.wav` -> 126 (preferred over 91, was 91)

**Impact on existing libraries**: any library with affected filenames
needs a `--rebuild` to pick up the corrected BPM column. The audit
query "samples whose stored BPM disagrees with librosa BPM" will surface
every misfire once features are populated.

**Status**: queued for v0.5.1 with `--discover`.

## Workflow gaps

### `with_features: 0` blocks similarity-style use cases

**Observation**: across three registered libraries (big_pack 32058,
moonboy 85, hypnotize 35) every library has `with_features: 0`. None
of the librosa-backed analysis tools work without that data.

**Implication**: a fresh user installs acidcat, runs `acidcat index DIR
--label NAME` once per library, and finds that `find_similar` returns
an error on every call. The fix instruction is "run `reindex_features`,
which takes hours" — that's a usability cliff.

**Possible improvements**:
- Make `acidcat index DIR --features` more discoverable (mention in the
  default no-args help).
- Auto-extract features for files with stored BPM/key during indexing,
  not just when `--features` is passed. Hidden cost vs. value trade-off.
- Background extraction: `acidcat index --extract-features-background`
  spawns a daemon that fills in features as the user works. Probably
  too clever for v0.5.1.
- Document the trade-off explicitly: "skip --features for fast indexing;
  add it later via `reindex_features` when you want similarity search."

**Status**: documented, no code change planned. Worth a README note.

### Cold-start cost on librosa makes ad-hoc detect_bpm_key painful

**Observation**: first librosa import takes ~60s on Windows due to
numba JIT compilation. Subsequent calls in the same Python process
warm. But each MCP call spawns a fresh process, so every "spot-check
this BPM" call from Claude pays the cold-start cost.

**Implication**: the LLM correctly avoids `detect_bpm_key` per the
SLOW prefix, but when the user explicitly asks "what's the actual
BPM of this file?", the answer takes a minute. Feels broken even when
working as designed.

**Possible improvements**:
- The MCP server could keep a persistent librosa-warmed worker around
  (subprocess that imports librosa, runs a tiny analyze on silence,
  then idles waiting for work). First MCP call after server start
  pays cold-start; subsequent are fast.
- Trade-off: complicates the server, adds memory cost (~500MB resident)
  the entire Claude Desktop session.

**Status**: not addressed in v0.5.1. Document the cost in the tool
description so the LLM and the user both know what to expect.

### Slow external-disk reads blow the 60s MCP timeout on `analyze_sample`

**Observation**: ad-hoc `analyze_sample` calls against files on the
G: drive (external/probably-spinning, holds big_pack 32058 files)
exceeded the 60s MCP client timeout. The LLM gracefully fell back to a
sample on a local SSD (Hypnotize pack on C:) and returned a full
analysis. The G: file was a single-digit-MB kick that should not take
60s to read on a fast disk.

**Cause**: cold-start librosa import (30-60s) + slow first read from
G: drive + actual feature extraction = total budget exceeded. C: drive
samples succeed because the disk read is fast enough that the JIT
overhead alone fits inside 60s.

**Implication**: at the user's scale (1.2M samples, mostly on external
drives), `analyze_sample` is unreliable for spot-checking. Same goes
for any future `reindex_features` run: a 32k-file extraction on G:
isn't going to complete inside a single `reindex_features` MCP call,
and the LLM will perceive it as broken.

**Possible improvements** (in order of effort):
1. **Bump the MCP server's per-call timeout signaling**. The 60s ceiling
   is the MCP client's, not ours, but we can split work into smaller
   resumable chunks. `reindex_features` already has a `limit` param;
   the LLM can run it in 100-file chunks and check `remaining_unprocessed`.
   `analyze_sample` is a single-file call so this doesn't help it.
2. **Pre-warm librosa worker** (already in the previous note). Removes
   the JIT cold-start from every analysis call. Probably the highest
   leverage fix.
3. **Local SSD scratch directory**. `acidcat config --cache-dir
   /local/path` plus a "warm cache" command that copies a library's
   audio to local storage before analysis. Heavy. Probably overkill
   unless the user does this routinely.
4. **Background batch extraction during idle**. The ` reindex_features`
   command runs as a daemon, processes one file at a time, persists
   progress, can be stopped/resumed. Nicer UX, more code.

**Status**: option 2 (worker pre-warm) plus making `reindex_features`
chunkable from the LLM's side feels like the right v0.5.1 scope.
Option 4 is v0.6 material.

### `reindex_features` on big_pack: throughput vs MCP timeout

**Observation**: chunked `reindex_features` runs (limit=20) on big_pack
(G: drive) hit the 60s MCP timeout cleanly. Each chunk completed
exactly 20 files before timing out, work was durably committed via
the per-library `conn.commit()` inside `reindex_features`.

Throughput: ~3s/file on G:. Cold librosa import accounts for ~30-50s
of the first call's budget; on warm subsequent calls within the same
MCP request, real per-file cost is closer to ~1s. The big_pack run is
mostly bottlenecked by spawning a fresh process per MCP call (each
one repaying the JIT cold start).

At 3s/file across 32058 audio files, full extraction = ~25-30 hours
of wall clock. Going through MCP at 20 files per call = ~1500 calls.
Not viable interactively.

**Architectural wins from this finding**:
1. **Per-batch commit pattern is correct.** Work persists across
   asyncio task cancellation. No data loss when MCP times out.
2. **The cost-prefix scheme correctly steered the LLM to push back
   on `reindex_features` before firing it on 32k files.** It chose
   limit=20 as a measurement chunk first, surfaced the math, asked
   for direction. That's the SLOW prefix doing its job.

**Engineering needed for v0.5.1 to make big_pack-scale extraction
viable**:

1. **Pre-warmed librosa worker** (already noted above): cuts ~30-50s
   off the first call of every MCP request. Probably the highest
   leverage single change. Brings effective per-file cost on G: from
   ~3s down to ~1s (~80% improvement on cold-start-dominated chunks).

2. **Periodic commits in the CLI walker.** `commands/index.py::run`
   currently calls `conn.commit()` once at the very end. A 25-hour
   `acidcat index --rebuild --features` run that gets interrupted
   loses everything. Need to commit every N files (suggest 100) for
   long-running unattended jobs.

3. **Dedicated `acidcat features LABEL [--resume]` CLI command.**
   Walks `samples` table for the given library, runs feature extraction
   on rows lacking features, commits per-file, can be interrupted
   and resumed. No directory walk, no parser dispatch. Cleanly
   separates "metadata indexing" from "audio analysis." Maps 1:1 to
   the existing `reindex_features` MCP tool.

**User-facing recommendation for testing v0.5.0**: skip big_pack
feature extraction until v0.5.1 ships items 1-3 above. Validate the
feature pipeline end-to-end on hypnotize (35 files) and moonboy (85
files) on local SSD instead. That's enough to confirm `find_similar`
works, that features persist, that `population` counts come back
correctly, etc. The architecture is the thing under test, not the
hardware throughput on a single user's external drive.

## What worked correctly

### Per-library DBs at registry-tracked paths

big_pack lives at `G:/DATAz/SampleZ/11111 overheat` (external drive,
different filesystem) but its DB lives at
`C:/Users/joshr/.acidcat/libraries/big_pack_33a89878.db` on C:. The
central-default storage decision held: drive separation didn't break
anything, registry resolves db_path correctly, queries fan out fine.

### LLM cost-prefix awareness

When asked "find samples whose librosa BPM disagrees with stored BPM",
the LLM (via MCP) correctly:
1. Checked `index_stats` first
2. Saw `with_features: 0`
3. Pushed back with "the premise can't be checked yet" rather than
   firing `reindex_features` on 32k files

That's the cost-prefix scheme working as designed. The SLOW marker
combined with "Only call when the user explicitly asks" in the
description prevented an accidental hours-long extraction.

### `analyze_sample` returns a feature vector that the LLM can interpret meaningfully

**Observation**: tested with `PL_Hypnotize_03_126_Kick.wav`. The LLM
ran `analyze_sample`, got the 50+ field feature vector back, and
produced a substantive read of the spectral content: identified
the saturated chroma as the A# tonal body, called out the std/mean
ratio on RMS as the transient signature, flagged the high spectral
contrast (13.4 dB) as evidence of a layered tuned kick, noted the
monotonic MFCC decay as evidence of clean processing.

This is the kind of analysis you can't get from looking at the raw
numbers; it requires knowing what each feature means and how they
combine. The MCP tool surface plus the cost-prefix scheme created
the right conditions for this.

**Implication**: `analyze_sample` is the tool worth keeping in the
SLOW tier. The cost is fair for the value when the file is on a fast
disk. Only pain point is the timeout issue above.

### library_label propagation

Every tool result that returns sample rows now includes `library_label`.
Verified during MCP testing across `search_samples`, `get_sample`,
`locate_sample`, `find_compatible`, and write tools (`tag_sample`,
`describe_sample`).

### Legacy v0.4 DB ignored cleanly

`~/.acidcat/index.db` from earlier testing is still on disk; the v0.5
indexer prints the one-line warning and otherwise leaves it alone.
Confirmed during the big_pack indexing run.

## Findings from end-to-end pipeline test (2026-04-27)

### Pipeline validates: register -> index -> features -> similarity

Bulk-registered all 15 sub-folders of `sample_packs/` plus a re-attach of
the existing two (MOONBOY, PL_Hypnotize_03_126_A#). All registrations
succeeded, idempotent on the second pass (path-hash detected the
existing registration and re-attached). Then `reindex` with
`with_features=true` on `808 Collection` (30 files, local SSD) cleanly
populated samples + features in well under the 60s MCP timeout.
`find_similar` against the longest-duration 808 fanned out across all
libraries with features (population: 50 = 30 from 808 Collection + 20
from big_pack from earlier failed runs).

That's the full vertical: registry mutation -> fan-out indexing ->
feature extraction -> cross-library similarity. All tools landed in the
right per-library DB. `library_label` propagated correctly in
`find_similar` result rows.

### `list_libraries` reports `sample_count: null` for re-attached DBs

**Bug**. When a library is `forget_library`'d and then re-registered,
its DB file still exists on disk with prior data. The new registration
correctly re-attaches (because the path hash is the same, so
`db_path` matches), but the registry's cached `sample_count` and
`last_indexed_at` are NULL until `reindex` runs.

For users this is misleading: "I registered this library with 35
samples in it, why does `list_libraries` say null?"

**Fix**: on `register_library`, if the target DB file already exists,
inspect it and populate `sample_count` / `feature_count` /
`last_indexed_at` from the existing DB before returning. Two extra
SELECTs at registration time. Easy.

**Status**: queued for v0.5.1.

### `find_similar` score compression at the top of the distribution

**Observation**: a `find_similar` call on `Cymatics - 808 Internet (C).wav`
returned the four sibling Cymatics 808s (same pack, same key, similar
duration) all in the 0.995-0.999 cosine range. Spread across rank 1-4
is 0.005, which is below the noise floor of any meaningful "this is
more similar than that" comparison.

The score compression is real: same-pack samples processed through
the same Cymatics signal chain produce nearly-identical aggregated
feature vectors. Cosine doesn't have headroom to discriminate inside
that cluster.

**Implications for the product**:
- For "find me more like this" creative crate-digging, the ranking
  inside a same-pack cluster is essentially arbitrary.
- For cross-pack comparisons, the absolute score is interpretable
  (0.99 vs 0.85), but within-pack ranking is not.

**Fix possibilities** (range from cheap to expensive):
1. Display percentile rank against the population, not raw cosine,
   so the user reads "rank 4 of 50, 92nd percentile" rather than
   "0.9922 vs 0.9999."
2. Subtract the population mean from each score before display, to
   surface relative-to-baseline rather than absolute.
3. Weight chroma + tonnetz higher than MFCC means when the target
   is tonal (high chroma_mean indicates tonal content). Brings pitch
   content into the discrimination layer.
4. Add temporal variance features (centroid_delta, MFCC trajectories)
   so "evolving noise" and "stable tone" don't cosine-collapse.

**Status**: option 1 is a v0.5.1 candidate (cheap, big UX win).
Options 3-4 are real feature-engineering work, probably v0.6.

### `find_similar` cross-library "sub-heavy long-tail" false positive

**Observation**: alongside the four sensible Cymatics 808 matches, the
fifth result was `91V_SBH_130_drum_fill_build_up_knight.wav` from
big_pack at sim 0.9922. That's a 7.4s bass house drum build-up: noisy,
percussion-heavy, no stable pitched fundamental. Sonically nothing
like a tonal 808.

The shared signal: both clips have heavy sub content, low spectral
centroid, large negative MFCC1 (bass-dominant tilt), high spectral
contrast. The aggregated means flatten the temporal evolution that
distinguishes "evolving noise" from "stable pitched body."

**This is the same family of issue as the `find_compatible` kind
filter.** When `find_compatible` defaults to filtering by `kind`
(loop vs one-shot inferred from acid_beats + duration), it avoids
similar mismatches there. `find_similar` doesn't do that filter; it
just returns top-N by cosine.

**Fix**: add a `kind_filter: bool` (default `true`) to `find_similar`,
mirroring `find_compatible`. When true, results are constrained to
the target's inferred kind. The 7.4s build-up wouldn't have surfaced
because target's kind is "one_shot" (0.4s, no acid_beats) and the
build-up's kind is "loop" (>2s).

**Status**: high-leverage v0.5.1 fix. Three lines of where-clause
change in the find_similar handler.

### `reindex with_features=true` is the right pattern for fresh libraries

**Observation**: the LLM noticed that running `reindex_features` on a
freshly-registered library would no-op because the samples table is
empty. Folded both passes into one `reindex(with_features=true)`
call. That worked cleanly.

**Implication**: documentation should make this pattern explicit.
"For a freshly-registered library, use reindex with_features=true.
For an already-indexed library where you just want to add features,
use reindex_features."

**Status**: README clarification + tool description tweak.

### LLM tool selection was correct throughout the pipeline

Across this session the LLM:
- Picked the right tool at each step
- Used metadata-only tools when possible
- Reached for SLOW tools only when the task required them
- Caught the "empty index" trap and fused two tool calls into one
  smarter call
- Provided substantive interpretation of feature vectors and called
  out a real similarity false-positive without being prompted

The cost-prefix scheme is doing its job. No tool description rewrites
needed based on this session.

## Things to watch for as testing continues

- **Path normalization on cross-drive paths**: G: drive paths normalize
  to `G:/DATAz/...` form. Verify nothing in the registry / query layer
  has a hardcoded assumption that paths start with `C:/`.
- **Filename regex collisions**: producer pack naming conventions vary
  wildly. Expect more parser misfires beyond the `91V` pattern. Log
  them all.
- **Tool selection failures**: any time the LLM picks the wrong tool
  (e.g. `find_similar` when `search_samples` would have worked), the
  description text needs work.
- **Query result quality**: when `find_compatible` returns harmonically
  valid but stylistically wrong samples (e.g. a kalimba loop returning
  trap drums in the same key), that's a "what other signal do we need"
  prompt, not a bug.
